文脈上、あなたの文は誤変換で、「**有線では正常、無線で悪化する**」として解析しました。

## 結論

公開情報、ZMK 上流、moNa2 の設定、PMW3610 ドライバのソースを突き合わせると、**クリティカルな根本原因はセンサー故障ではなく、`PMW3610 → ZMK pointing → BLE HID` 経路の設計上のミスマッチ**です。さらに moNa2 側では、そのミスマッチを緩和するためにドライバが用意している `CONFIG_PMW3610_REPORT_INTERVAL_MIN` が `mona2_r.conf` でコメントアウトされており、ドライバ既定値の `0` が使われます。ドライバ自身はこの設定を「**over the air の HID queue のために入力報告を遅くする**」用途だと明記しており、ZMK v0.3 系の BLE 側にはマウス report queue サイズ 20 と、接続間隔 6–12（1.25ms 単位なので 7.5–15ms）という制約があります。つまり、**高頻度なトラボ入力を、無整形のまま BLE の有限キューと接続間隔スケジューリングに流し込んでいる**のが本丸です。USB ではこの制約が消えるので、症状が出ない説明がつきます。 ([GitHub][1])

## なぜそう断定できるか

まず、**moNa2 のトラックボールは右手側の central 側に載っています**。`mona2_r.overlay` では右手側に `trackball_central` が定義され、`Kconfig.defconfig` では `SHIELD_MONA2_R` が `ZMK_SPLIT_ROLE_CENTRAL` です。したがって、moNa2 の右手トラボは「左半分→右半分」の split BLE hop を経由していません。ZMK の split ドキュメントが説明している平均 3.75ms / worst 7.5ms の増分は peripheral 起点イベントの話であり、**moNa2 右手トラボの主因ではない**と切り分けられます。 ([GitHub][2])

次に、**現行ファームウェアの依存関係がかなり重要**です。moNa2 v2 の `west.yml` は、ZMK 本体を `v0.3-branch`、PMW3610 を `badjeff/zmk-pmw3610-driver` の `zmk-0.3` に固定しています。つまり、moNa2 は「ZMK の BLE ポインタ系がまだ荒れやすい時期の分岐 + 外部ドライバ」の組み合わせで動いています。これは単なる最新 mainline 構成ではありません。 ([GitHub][1])

ソースコード上の“煙が出ている箇所”は、やはり `PMW3610_REPORT_INTERVAL_MIN` です。ドライバはこの値が 0 より大きい場合だけ、`dx/dy` を蓄積し、一定間隔未満では report を返さず、間隔が来たタイミングで `input_report(..., K_NO_WAIT)` を送ります。ところが moNa2 の `mona2_r.conf` では `# CONFIG_PMW3610_REPORT_INTERVAL_MIN=12` がコメントアウトされたままです。しかもドライバ Kconfig の既定値は 0 です。したがって現在の moNa2 は、**BLE 用に用意されている入力整形を無効のまま**、センサーイベントを ZMK 側へ渡している構成です。これは moNa2 固有の増幅点です。 ([GitHub][3])

さらに重要なのは、moNa2 の `CONFIG_BT_PERIPHERAL_PREF_MIN_INT=6` / `MAX_INT=12` が、ZMK v0.3 側の既定値と**同じ**だという点です。つまり、moNa2 だけが異常に悪い BLE 間隔を選んでいるわけではありません。問題は値の異常さではなく、**BLE マウス経路そのものが有限 queue と connection interval scheduling を持つこと**です。7.5–15ms 単位でイベントが量子化される経路に対して、PMW3610 の細かい移動を無整形で流せば、親指トラボのような高頻度・低振幅入力で引っかかりや微妙な飛びが出やすい、と考えるのが自然です。 ([GitHub][3])

## コミュニティ側の観測も一致している

上流の ZMK issue #2957 では、**「USB では smooth scrolling は正常だが、BLE だとかなり跳ぶ」**と報告されています。moNa2 の右手 overlay には `zip_xy_to_scroll_mapper` を使う scroll layer も定義されているので、あなたの「反応が悪い」がスクロールを含むなら、これはかなり直接的に同じ経路の問題です。つまり、moNa2 だけの孤立障害ではなく、**ZMK/BLE のポインタ出力系そのものに USB との差がある**ことが上流でも確認されています。 ([GitHub][4])

別の ZMK 系無線トラボキーボード `mtk64ebt` の公開資料でも、**無線時は約 70Hz、USB 時は約 220Hz**、さらにドングル使用時は **約 110–220Hz** 相当と明記されています。これはまさに「無線だとトラボが鈍く感じる」現象の公開実測例です。moNa2 と同一機種ではありませんが、同じ ZMK/BLE 系で、**USB と BLE の体感差が設計上出る**ことの強い傍証です。 ([GitHub][5])

ただし、ここは誤解しやすいので補足します。ZMK の dongle ドキュメントは、USB hop への置き換えで全体平均 latency が下がるケースを説明していますが、**former central には約 1ms の追加 hop** が乗るとも書いています。moNa2 のトラボは central 側なので、dongle は「BLE host hop が効いている」ことの証拠にはなっても、**moNa2 の右手トラボに対する決定打とまでは言えません**。この点は、よくある「dongle で全部解決」という話をそのまま当てはめると誤ります。 ([ZMK Firmware][6])

## 根本原因ではないもの

`badjeff/zmk-pmw3610-driver` には、**スリープ復帰後に `SWAP_XY` / `INVERT_X` / `INVERT_Y` が効かなくなる** open issue もあります。もし症状に「復帰直後だけ方向がおかしい」「たまに軸反転設定が消えたようになる」が含まれるなら、これは別件の二次バグとして乗っている可能性があります。ですが、これは**一般的な“無線時だけ追従が悪い”主因ではありません**。 ([GitHub][7])

また、`mona2_r.overlay` の `cpi = <600>` は操作感には影響しますが、**CPI は感度の話**であって、USB では正常なのに BLE だけ悪化する説明にはなりません。wired / wireless 差を説明しているのは、やはり transport 側です。 ([GitHub][2])

## 最終判定

公開情報だけからの最終判定は、かなり明確です。**moNa2 の無線トラックボール問題の本質は、PMW3610 センサー不良でも、左右 split 間通信でもなく、ZMK v0.3 系 BLE マウス出力の帯域・スケジューリング制約に対して、moNa2 側が `PMW3610_REPORT_INTERVAL_MIN` による over-the-air 向け report shaping を無効のまま使っていること**です。言い換えると、**根本原因は「BLE マウス経路の構造的制約」+「その制約に合わせるべき入力整形が無効」**の組み合わせです。したがって、CPI 調整や左右通信だけを疑っても本丸は外れますし、設定を少し触って改善しても、**現行構成のまま USB 同等の追従性を完全に得るのは難しい**、というのが公開情報からの結論です。 ([GitHub][3])

結論から言います。

**v2 で最初に直すべきなのは、`mona2_r.conf` と `mona2_r.overlay` の右手側設定です。**
この v2 リポジトリは、`config/west.yml` で **ZMK 本体を `v0.3-branch`、PMW3610 ドライバを `badjeff/zmk-pmw3610-driver` の `zmk-0.3`** に固定しています。さらに右手側 `mona2_r` は `ZMK_SPLIT_ROLE_CENTRAL` で、右手 overlay 側に `trackball_central` が定義されているため、**修正対象は右手側に集中**します。ここは重要で、あなたが言及した「shakushakupanda 系の独自ドライバ」が **この v2 で実際に使われているわけではありません**。その系統の fork には shakushakupanda のコミットがありますが、README 上は別の ZMK ブランチ `feat/pointers-move-scroll` を前提にしており、moNa2 v2 の現行スタックとは別物です。 ([GitHub][1])

根本原因に対して、v2 で一番効く修正は次です。
`mona2_r.conf` では **`CONFIG_PMW3610_REPORT_INTERVAL_MIN=12` がコメントアウトされたまま**で、依存している badjeff の `zmk-0.3` ドライバではこの値の既定値は **0** です。しかもその Kconfig ヘルプには、この設定が **“hid queue over the air” 向けに入力報告を遅くするため**のものだと明記されています。実装上も、この値が 0 より大きいときだけ `dx/dy` を蓄積して一定間隔ごとに report し、そうでなければ高頻度でイベントを流します。一方 ZMK v0.3 側では BLE の **mouse report queue の既定値が 20**、接続パラメータ既定は **min 6 / max 12 / latency 30** です。つまり、**今の v2 は “BLE 向けのレート整形が無効” のまま、“有限キューの BLE” に流している**状態です。まずここを直すべきです。 ([GitHub][2])

## まず入れるべき修正

### 1. `config/mona2_r.conf`

```diff
diff --git a/config/mona2_r.conf b/config/mona2_r.conf
@@
-# CONFIG_PMW3610_REPORT_INTERVAL_MIN=12
+CONFIG_PMW3610_REPORT_INTERVAL_MIN=12
 # CONFIG_PMW3610_LOG_LEVEL_DBG=y
 # CONFIG_PMW3610_INIT_POWER_UP_EXTRA_DELAY_MS=300 // <--see Troubleshooting

@@
 CONFIG_BT_PERIPHERAL_PREF_MIN_INT=6
-CONFIG_BT_PERIPHERAL_PREF_MAX_INT=12
+CONFIG_BT_PERIPHERAL_PREF_MAX_INT=6
+CONFIG_ZMK_BLE_MOUSE_REPORT_QUEUE_SIZE=40
+
+# まだ無線で「動き始めが鈍い」なら追加
+# CONFIG_BT_PERIPHERAL_PREF_LATENCY=0
```

このパッチの意味は明確です。

`CONFIG_PMW3610_REPORT_INTERVAL_MIN=12` は、**今の v2 で最優先**です。依存ドライバ自身が BLE 向けの report 整形用途として説明しており、`8ms` は **USB を使う場合の鋭い反応向け**と書かれています。したがって **BLE 優先なら 12ms から始める**のが妥当です。コミュニティ側でも `12` を起点に調整している例があり、数字を下げるとより細かくなる一方で負荷や消費は増える、という理解で一致しています。 ([GitHub][3])

`CONFIG_BT_PERIPHERAL_PREF_MAX_INT=6` は、今の v2 が **`min=6 / max=12`** になっているので、これを **`6 / 6` に寄せる**修正です。Zephyr の BLE 接続パラメータでは `int_min` / `int_max` は **1.25ms 単位**なので、`6` は **7.5ms** です。今は 7.5–15ms の可変レンジを許しているため、**報告タイミングのブレ**が出やすい。`6/6` に寄せれば、少なくともキーボード側は **最短側を強く要求**できます。これは “要求” であって host が必ず採用する保証はありませんが、設定としては合理的です。 ([GitHub][2])

`CONFIG_ZMK_BLE_MOUSE_REPORT_QUEUE_SIZE=40` は、ZMK v0.3 の既定値 **20** を増やす修正です。これは **根治ではなくバッファ拡張**ですが、PMW3610 からの移動 burst を host 側 BLE スケジューリングが吸収しきれない瞬間の **取りこぼし耐性**を上げます。40 ならまだ現実的です。64 でも動く可能性はありますが、私はまず 40 を勧めます。 ([GitHub][4])

`CONFIG_BT_PERIPHERAL_PREF_LATENCY=0` は **第二段階**です。ZMK v0.3 の既定値は `30` で、BLE パラメータ上の `latency` は接続イベントの扱いに影響します。**レスポンス優先・バッテリー不利**の設定なので、まずはコメントのまま置き、Patch 1 を入れても「少しアイドルした後の最初の動きだけ鈍い」なら有効化する、という順序がいいです。 ([GitHub][4])

### 推奨チューニング範囲

`REPORT_INTERVAL_MIN` はこう触るのがいいです。

* **12ms**: BLE での第一候補
* **10ms**: 12ms でまだ鈍いが、取りこぼしは減った場合
* **8ms**: USB 寄り。BLE 常用では最初から勧めない

これはドライバの説明とも整合します。 ([GitHub][3])

## 次に入れるべき修正

### 2. `boards/shields/mona2/mona2_r.overlay`

```diff
diff --git a/boards/shields/mona2/mona2_r.overlay b/boards/shields/mona2/mona2_r.overlay
@@
         trackball_central: trackball_central@0 {
             status = "okay";
             compatible = "pixart,pmw3610";
             reg = <0>;
             spi-max-frequency = <2000000>;
             irq-gpios = <&gpio0 2 (GPIO_ACTIVE_LOW | GPIO_PULL_UP)>;
             cpi = <600>;
+            force-awake;
             //swap-xy;
             //invert-x;
             //invert-y;
             evt-type = <INPUT_EV_REL>;
             x-input-code = <INPUT_REL_X>;
             y-input-code = <INPUT_REL_Y>;
         };
```

現行 v2 の `trackball_central` ノードには **`force-awake` が入っていません**。同じ badjeff 系ドライバの README では `force-awake` が **ACTIVE 中はセンサーを起こしたままにする**用途として説明されており、`force-awake-4ms-mode` は **250Hz の direct USB 用**だと明記されています。
したがって、**BLE で困っているなら `force-awake` は候補、`force-awake-4ms-mode` は入れない**、これが正しい順序です。4ms mode は BLE 側のキューと接続間隔に対してむしろ圧を上げます。 ([GitHub][5])

この修正は、**アイドル後の最初の動きが鈍い**、**小さい動きの立ち上がりが眠そう**という症状に効く可能性があります。
ただし、これは **消費電力とのトレードオフ**です。だから順番としては、**先に `REPORT_INTERVAL_MIN` と BLE 接続設定**、その後に `force-awake` です。

## スクロールが「遅い / 少ない」なら別件でここも直す

v2 の right overlay では、スクロールレイヤで

```dts
<&zip_xy_to_scroll_mapper>
<&zip_scroll_scaler 1 5>;
```

が入っています。ZMK の scaler ドキュメントでは、この 2 つの引数は **multiplier / divisor** です。つまり `1 5` は **1/5 倍**です。ところがコード内コメントは **「1/4倍」**になっており、**コメントと実装が一致していません**。
したがって、あなたの「反応が悪い」が **スクロール量が少ない**ことを指しているなら、ここは transport 問題とは別に調整対象です。たとえば次です。 ([GitHub][5])

```diff
diff --git a/boards/shields/mona2/mona2_r.overlay b/boards/shields/mona2/mona2_r.overlay
@@
-  <&zip_scroll_scaler 1 5>;
+  <&zip_scroll_scaler 1 4>;
```

あるいはもう少し速くしたいなら `1 3` です。
ただし、これは **BLE ラグの修正ではなく、スクロール量の体感調整**です。根本修正ではありません。

## それでも直らない場合の上位修正

ここまで入れても不満が残るなら、次は **スタック更新**です。
ただしこれは “設定調整” ではなく **依存更新** です。

badjeff の現行 README では、`main` / `zmk-0.4` 系では **互換文字列が `pixart,pmw3610-alt` に変わり、設定名も `CONFIG_PMW3610_ALT_*` へ変更**される breaking change が明記されています。また、その系統では **sampling rate と reporting rate を分離**する設計が明確化されています。つまり、**現行 v2 の `v0.3-branch + zmk-0.3` よりは筋の良い土台**です。 ([GitHub][6])

その advanced path はこうです。

1. `config/west.yml` の ZMK と PMW3610 を newer stack に上げる
2. overlay の `compatible = "pixart,pmw3610"` を `pixart,pmw3610-alt` に変更
3. `.conf` の `CONFIG_PMW3610_*` を `CONFIG_PMW3610_ALT_*` に変更
4. 右手だけでなく **左右両方**を再ビルド・再書き込みする

ただし、これは v2 の他モジュールとの整合も見る必要があるので、**最初の一手ではありません**。
まずは上の A/B パッチを入れるのが正解です。 ([GitHub][1])

最後に一点だけ実務上の補足をします。
**今回の hotfix は right half 専用です。** `build.yaml` でも `mona2_r` と `mona2_l` は別ビルドになっており、トラボ本体は右手側にあります。したがって **A/B パッチだけなら右手側ファームだけを更新して検証できます**。`west.yml` を触る advanced path に進むときだけ、左右両方を更新してください。 ([GitHub][7])

まず前提です。**この行について開発者本人が「なぜコメントアウトしたか」を明言した一次ソースは見つかっていません。** `mona2_r.conf` の履歴に見えるコミット名も `update moNa2.0`、`ZMK Studio対応`、`update_configfile_READMEfile`、`Update mona2_r.conf` という簡潔なものです。なので、以下は**コミット履歴・blame・依存ドライバ README・作者の記事からの設計判断プロファイル**です。 ([GitHub][1])

## 結論

私の見立てでは、**「その設定だけを危険視して切った」のではなく、v2 移行時に low-level PMW 調整ブロック全体を“オプションのまま残した”**、が最有力です。しかもその判断は、**BLE/USB で体感が割れる transport 依存のチューニングを、公開デフォルトに焼き込まない**というメンテナ視点と整合します。 ([GitHub][2])

## 判断の内訳

### 1. これは「後から無効化した」のではなく、**最初からコメントのまま置かれた PMW オプション群**です。確度は高いです。

`mona2_r.conf` では `INVERT_X`、`INVERT_Y`、`SWAP_XY`、`REPORT_INTERVAL_MIN`、`LOG_LEVEL_DBG`、`INIT_POWER_UP_EXTRA_DELAY_MS` がまとめてコメントアウトされています。blame ではこのブロック全体が **2025-08-14 の `update moNa2.0`** に由来し、後続の 2025-09-24 のコミットは LED、ZMK Studio、BT 設定側を触っているだけで、この PMW ブロックを「有効→無効」に戻した痕跡はありません。つまり、意図としては **“12ms を試して問題があったので消した”** より、**“PMW の低レベル調整はテンプレ状態で残した”** と読む方が自然です。 ([GitHub][2])

### 2. その PMW ブロックは、**badjeff ドライバ由来の“参考設定の雛形”を持ち込んだ**可能性が高いです。これも確度は高いです。

v2 の `west.yml` は ZMK を `v0.3-branch`、PMW3610 ドライバを `badjeff/zmk-pmw3610-driver` の `zmk-0.3` に固定しています。しかも、そのドライバ README には `# CONFIG_PMW3610_REPORT_INTERVAL_MIN=12` を含む**同型のコメント付き設定ブロック**が載っています。さらに README は `CONFIG_PMW3610_SWAP_XY` / `INVERT_*` を **device tree 属性へ置き換え済み**だと説明しているのに、moNa2 側の `.conf` にはそれらがコメントのまま残っています。これはかなり重要で、**実運用で有効にする前提の厳選設定**というより、**移行時に参照用・上級者向けノブとして残したブロック**と見る方が整合的です。作者自身も v2 更新記事で、ファームウェアを ZMK v0.3 向けに作り直し、PMW3610 は badjeff 系ドライバへ切り替えたと書いています。 ([GitHub][3])

### 3. コメントアウトの実質理由は、**`REPORT_INTERVAL_MIN` が“無償の改善”ではなく、BLE 向けの体感トレードオフ設定**だからだと思われます。確度は中〜高です。

依存している `zmk-0.3` ドライバの Kconfig では、この設定は **default 0** で、説明文は **「hid queue over the air のために入力報告を遅くする」**、さらに **「USB only なら 8ms に下げると鋭い反応」** です。ドライバ実装も、これを有効にすると `dx/dy` を蓄積し、一定間隔未満では report せず、間隔が来たらまとめて `input_report()` する動作になっています。つまりこれは、**BLE での queue 詰まりやバーストを和らげる代わりに、ポインタ感触に介入する設定**です。公開ファームウェアのデフォルトとしては、開発者から見ると「改善」ではなく **“どの体感を標準にするか” の政策判断**になります。だから opt-in に留めた、という説明が最も筋が通ります。 ([GitHub][4])

### 4. 作者の設計思想とも一致します。**ワイヤレス前提・省電力寄り・高レベル側で調整**、というプロファイルです。確度は中です。

作者は ZMK を「**無線接続に最適化**」「**低消費電力設計**」として選んでいます。実際の `mona2_r.overlay` でも SPI の sleep 側 pinctrl に `low-power-enable` を入れており、一方で badjeff README の例にある `force-awake` / `force-awake-4ms-mode` は moNa2 では有効化していません。さらに v2 の解説記事では、マウス関連の処理はドライバ内部ではなく **ZMK 側の `input-processors`** で扱うように変更したと説明し、マウス動作のカスタマイズ対象として `mona2.dtsi` と `mona2_r.overlay` を案内しています。要するに、開発者視点では **低レイヤのセンサ timing をデフォルトでいじるより、上位レイヤで動作を決める** 方が v2 の設計に合っていたはずです。 ([note（ノート）][5])

### 5. その repo は、**“コメントを外して使う上級者向けオプション” を意図的に残す書き方**をしています。確度は中です。

作者は moNa2 の設定記事で、`//` はコメントアウトを意味し、**有効化したい場合は先頭の `//` を削除する**と明記しています。しかも同じ記事で、デフォルトのマウス動作を置きつつ、必要ならオプションを有効化して使う流れを説明しています。したがって、`# CONFIG_PMW3610_REPORT_INTERVAL_MIN=12` も「使うな」という意味より、**“必要なら分かる人が有効にするノブ”** と読む方が、作者の公開スタイルに一致します。 ([note（ノート）][6])

### 6. なお、**この判断を shakushakupanda に帰属させるのはズレています**。ここは修正しておきます。

`mona2_r.conf` の PMW コメントブロックは blame 上 **sayu-hub の 2025-08-14 `update moNa2.0`** に由来します。履歴上、`shakushakupanda` の `Update mona2_r.conf` はその後の 2025-09-24 で、実際に触っているのは後半の LED/Bluetooth 周辺です。さらに作者の振り返り記事でも、shakupan さんは主に **CAD・3D プリント・外装設計支援**の文脈で紹介されています。なので、この `REPORT_INTERVAL_MIN` をコメントのまま残した設計判断は、現状の公開情報では **sayu-hub 側の判断**と見るべきです。 ([GitHub][7])

## 最終的な人物像としての回答

開発者の視点に立つと、たぶんこうです。
**「v2 は ZMK v0.3 + badjeff ドライバへの大移行。まずは wireless 前提で広く破綻しない保守的デフォルトを出す。低レベル PMW ノブは残すが、BLE でしか効きづらい体感トレードオフを標準値としては押し付けない。必要な人だけコメントを外して調整してほしい。」**
私はこれが、このコメントアウトの一番自然な説明だと見ています。 ([note（ノート）][6])
